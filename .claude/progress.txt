# Log de Aprendizados por Plano

## Plano 7 – Frontend: Biblioteca de Roteiros / Grid Netflix (concluído em 2026-05-13)

### O que foi feito
- `useRoteiros.js`: hook com fetch autenticado (`credentials: 'include'`), debounce 300ms só na busca, suporte a filtros múltiplos (temas como array com `temas[]`), estado de carregando/erro/dados
- `RoteiroCard.jsx`: card poster aspect-ratio 2:3 com overlay de hover (gradiente, sinopse, tag, botão "Ver roteiro"), rodapé fixo para mobile (oculto em sm+), fallback visual para sem capa
- `RoteiroGrid.jsx`: grid responsivo 2/3/4 colunas, 8 skeletons no loading, mensagem de erro, EmptyState quando vazio
- `SearchBar.jsx`, `SortSelector.jsx`, `FilterBar.jsx`: filtros compostos com múltipla seleção de temas (toggle visual), dropdown de mês derivado dos roteiros, dropdown de tipo fixo, botão "Limpar filtros" condicional
- `BibliotecaPage.jsx`: integração completa com sticky bar de controles (bg-page, z-10) + área scrollável do grid
- `CarrosselTematico.jsx`: scroll horizontal com botões prev/next visíveis no hover via `group/carrossel`, `.scrollbar-hide` adicionado ao `index.css`
- Setup de testes: vitest + @testing-library/react + jsdom; setupTests.js; vite.config.js atualizado; script "test" no package.json
- 11 testes passando: 4 em RoteiroCard, 4 em RoteiroGrid, 3 em BibliotecaPage

### Aprendizados e observações
- O componente `Tag` existente (do Plano 6) não tem prop `variante` — usa prop booleana `ativa` para estilo âmbar. Adaptar uso de `<Tag variante="tema">` para `<Tag ativa>`.
- Debounce no hook: usar `setTimeout`/`clearTimeout` dentro do `useEffect` com a busca como dependência; outros filtros (mês, tipo, temas, ordenação) não precisam de debounce — disparar imediatamente com `delay: 0`.
- No `RoteiroGrid`, temas foram derivados dos roteiros em tempo real (via `flatMap`) — evita hardcode e funciona com qualquer estrutura de dados da API.
- `group/carrossel` é a sintaxe de named group do Tailwind CSS v4 para hover em múltiplos elementos aninhados — permite `group-hover/carrossel:opacity-100` sem conflito com outros `group`.
- Em testes com `useNavigate`, mockar apenas `react-router-dom` com `{ useNavigate: () => vi.fn() }` é suficiente — não precisa de `MemoryRouter`.
- `screen.getAllByText()` (plural) evita falha quando o mesmo texto aparece em múltiplos elementos do DOM (overlay desktop + rodapé mobile do RoteiroCard).

### Aprendizados e observações
- Migrations com mesmo timestamp são ordenadas alfabeticamente → `admin_audit_log` viria antes de `admin_users` (FK failure). Solução: ajustar timestamps manualmente para garantir admin_users antes de admin_audit_log.
- O Laravel 12 não cria `config/cors.php` por padrão — é criado diretamente no arquivo.
- A session padrão do Laravel 12 já usa database driver; o arquivo de sessions padrão precisou ser removido pois a migration custom de magic link usa a mesma tabela `sessions`.
- O `npm run build` do frontend passou sem erros após mover para `frontend/` (154 pacotes, build em 259ms).
- `php artisan migrate --pretend` valida a syntax sem precisar de banco conectado — útil para CI.

## Plano 2 – Autenticação por Magic Link (concluído em 2026-05-13)

### O que foi feito
- Model `Sessao` com scopes `ativa()` e `expirada()` (tabela `sessions`)
- `LastlinkService`: wrapper da API Lastlink com `Cache::remember()`, fail-closed (retorna false em erros), `invalidarCache()` para webhooks
- `EmailService`: envio de magic link via `Mail::send()` com template HTML inline em português
- `MagicLinkService`: gera token raw (64 hex chars), armazena apenas SHA-256 no banco, invalida tokens anteriores não usados
- `SessaoService`: verificação de token, criação de cookie httpOnly (30 dias), leitura de sessão ativa, revogação
- Middleware `VerificarAssinaturaAtiva`: 401 sem cookie, 403 sem assinatura ativa, injeta `email_membro` no request
- Controllers `AuthController` e `LogoutController` (finos, apenas delegam para Services)
- `routes/api.php` criado e registrado no `bootstrap/app.php` com alias `auth.membro`
- 10 testes de feature passando (fluxo completo + casos de erro)

### Aprendizados e observações
- Laravel `postJson()` / `getJson()` nos testes não enviam cookies automaticamente — é necessário `->withCredentials()->withUnencryptedCookie('nome', $valor)` para simular sessão autenticada nos testes feature.
- Cookies criptografados pelo Laravel não são descriptografados nos testes a menos que o middleware `EncryptCookies` esteja presente no grupo de rotas — a rota API não o tem por padrão.
- `bootstrap/app.php` no Laravel 12 substitui `app/Http/Kernel.php` — middleware aliases são registrados via `$middleware->alias([])` no `withMiddleware()`.
- Token raw nunca deve ser armazenado — apenas o hash SHA-256. O token raw vai por e-mail, o hash fica no banco.
- Fail-closed no `LastlinkService` é crítico: qualquer falha na API externa deve negar o acesso (retorna `false`), não conceder.

## Plano 3 – Admin Backend: Auth + CRUD de Roteiros (concluído em 2026-05-13)

### O que foi feito
- Migration adicional para coluna `role` enum('admin','operator') na tabela `admin_users`
- Models `AdminUser`, `Roteiro` (com SoftDeletes e scopes) e `AdminAuditLog` criados
- `AdminAuthService` com login (email+senha), verificação 2FA TOTP e geração de secret (lib `spomky-labs/otphp` v11.4)
- Middleware `AutenticadoComoAdmin` com alias `auth.admin` em bootstrap/app.php
- `FileUploadService` para upload de capa (disco público) e PDF (disco privado)
- `AuditLogService` com constantes de ação e registro de ip_address em todo CRUD
- `RoteiroService` com CRUD completo: slug único com sufixo incremental, cálculo de status por data, filtros paginados
- FormRequests `LoginAdminRequest`, `CriarRoteiroRequest`, `AtualizarRoteiroRequest` com mensagens em português
- `AdminAuthController` (login, 2fa, logout) e `AdminRoteiroController` (index, store, show, update, destroy, despublicar)
- Command `roteiros:publicar-agendados` agendado a cada minuto via `routes/console.php`
- `AdminUserSeeder` idempotente (firstOrCreate) com admin padrão para setup inicial

### Aprendizados e observações
- `spomky-labs/otphp` v11.4 usa `TOTP::createFromSecret($secret)` (não `TOTP::create($secret)`) — API mudou entre versões.
- O cast `'password' => 'hashed'` no model AdminUser faz o hash automaticamente — passar a senha raw no seeder sem Hash::make() evita double-hashing.
- Laravel 12 registra o scheduler em `routes/console.php` via `Schedule::command()` (não em `app/Console/Kernel.php` que não existe mais).
- Pending token para 2FA armazenado no Cache com TTL de 5 min — ao validar, invalidar imediatamente (single-use) para evitar replay attack.
- SoftDeletes no model Roteiro: `delete()` faz soft delete; para hard delete usar `forceDelete()`. O comando de publicação só busca registros não deletados.
- `class_basename($entidade)` retorna nome curto da classe (ex: 'Roteiro') — ideal para coluna `entidade_tipo` no audit log sem namespace completo.

## Plano 4 – Webhooks Lastlink + Serviço de E-mail (concluído em 2026-05-13)

### O que foi feito
- Migration adicional para coluna `event_id` string nullable unique em `webhooks_log`
- Model `WebhookLog` com constantes de status, `jaProcessado()` estático e scope `pendentes`
- `WebhookController` com validação HMAC-SHA256, idempotência por `event_id`, registro rápido em `webhooks_log` e dispatch de job
- `ProcessaWebhookJob` (ShouldQueue, 3 tries, 60s backoff) — atualiza status para `processado` ou `erro`
- `ProcessaEventoService` com `match()` para 5 eventos: Purchase_Order_Confirmed, Subscription_Renewed, Subscription_Canceled, Payment_Refund, Payment_Chargeback
- `EmailService` estendido com `enviarBoasVindas()` e `enviarCancelamento()` usando Blade views
- 3 templates Blade em `resources/views/emails/` (magic-link, boas-vindas, cancelamento) com estilo cinematográfico
- Rota pública `POST /api/webhooks/lastlink` registrada sem middleware de autenticação
- `config/services.php` atualizado com chave `webhook_secret`
- 3 testes de feature passando (HMAC válido, HMAC inválido, idempotência)

### Aprendizados e observações
- "Invalidar sessão" com autenticação cookie-based sem server-side storage = invalida o cache Lastlink via `LastlinkService::invalidarCache()`. O middleware `VerificarAssinaturaAtiva` faz a verificação a cada request — ao invalidar o cache, a próxima requisição do usuário consulta a Lastlink e vê assinatura inativa.
- HMAC deve usar `hash_equals()` (comparação em tempo constante) para evitar timing attacks — nunca usar `===` direto.
- Job re-lança a exception após registrar o erro no banco — isso ativa o mecanismo de retry do Laravel Queue sem perder o estado.
- Em ambiente local/testing sem `LASTLINK_WEBHOOK_SECRET` configurado, o HMAC é bypassado para facilitar desenvolvimento — falha-aberta só em envs seguros, fail-closed em produção.
- `Queue::fake()` nos testes impede que jobs sejam de fato processados — usar `Queue::assertPushed()` para verificar dispatch sem efeitos colaterais.
